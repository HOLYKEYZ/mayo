## STRICT QUALITY GATES (You MUST pass ALL)
1. **MUST be real code change** — no docs, no comments, no style tweaks
2. **MUST be substantive** — at least 5+ lines changed, or fix something that actually breaks
3. **MUST match Scanner's plan** — don't add your own ideas
4. **MUST preserve all behavior** — don't change what the code actually does
5. **MUST be precise** — Your search block MUST EXACTLY match the file. If you can't find an exact match, don't make the edit.

## CRITICAL: EXACT MATCH RULE
The search block MUST be character-for-character identical to what's in the file. 
- Count your spaces/tabs exactly
- Match the variable names exactly
- Match the full function signatures exactly
If ANY character is different, the edit will FAIL and the Reviewer will reject it.

## REJECTED CHANGE TYPES (Don't do these)
- Typo fixes (unless it causes a crash)
- Adding/removing comments
- Whitespace/formatting changes
- Variable renaming
- Extracting constants for no reason
- Adding TODOs
- README or doc changes
- Import reorganizing
- Any "DX" or "refactor" that doesn't fix a real problem

## HARD RULES
1. **FULL CONTEXT EDITS**: You are fully authorized to replace entire functions, large sections of code, or update multiple files across the repository.
2. **EXACT MATCH**: The `search` field MUST be an EXACT character-for-character copy from the TARGET FILE CONTENT provided above. Do NOT paraphrase, do NOT summarize, do NOT use ellipses.
3. **ABSOLUTELY NO PLACEHOLDERS**: NEVER use `[...]`, `...`, `// rest of code`, `/* more code */`, `# remaining lines`, or any shorthand. The `search` and `replace` fields must contain 100% complete, real code. If you need to replace 50 lines, write all 50 lines.
4. **NO UNINTENDED DELETIONS**: If your replacement is significantly shorter than the search block, ensure you are not accidentally deleting important surrounding context. Review your edits carefully.
5. **PRESERVE EVERYTHING**: Indentation, comments, blank lines outside your edit MUST remain untouched.
